# Отчёт: Система знаний команды (Team Knowledge Base)

## 📋 Общая информация

**Название проекта:** Система знаний команды (Team Knowledge Base)  
**Технологии:** Flask, SQLite, gpt-4o-mini (proxyapi.ru), FAISS, sentence-transformers  

---

## 🎯 Ценность (для кого и зачем)

| Пользователь | Ценность |
|--------------|----------|
| **Новичок в команде** | Быстро находит правила, процессы и термины без помощи коллег (экономия 2-3 дня онбординга) |
| **Менеджер/оператор** | Получает готовые ответы для клиентов с цитатами-источниками (экономия 15-30 мин на поиск) |
| **Руководитель/тимлид** | Обеспечивает единую точку правды, доказуемость ответов и полный аудит действий |

**Проблема:** Информация раскидана по файлам/чатам/документам, поиск занимает время, опасно додумывать ответы.  
**Решение:** Система знаний с векторным поиском, LLM-ответами с источниками и честной ручной проверкой.  
**Результат:** Экономия ~80% времени на поиск, доказуемость ответов через цитаты, прозрачный аудит всех действий.

---

## 📝 3 сценария пользователя

1. **Загрузка документов** — администратор добавляет документы в базу знаний через веб-панель или API. Документы автоматически разбиваются на смысловые чанки (по 1000 символов с перекрытием 200) и векторизируются для семантического поиска.

2. **Поиск и ответ** — пользователь задаёт вопрос на естественном языке, система находит релевантные фрагменты через векторный поиск (FAISS), LLM формирует ответ с цитатами-источниками.

3. **Ручная проверка** — если данных недостаточно, система честно отвечает «Данных недостаточно», выставляет `needs_review=true`, фиксирует причину в аудите, помечает запись в истории для ручной обработки менеджером.

## 🤖 ИИ операция

### Схема JSON (строгий формат ответа LLM)
```json
{
  "answer": "string — текст ответа",
  "sources": [{"quote": "string — цитата из контекста"}],
  "confidence": "high | medium | low",
  "needs_review": true | false
}
```

### Параметры
- **Модель:** gpt-4o-mini через proxyapi.ru
- **Температура:** 0.1 (низкая для максимальной строгости)
- **max_tokens:** 1000
- **Попытки:** 3 на случай сбоя или некорректного JSON

### Когда нужна ручная проверка
- Источники не найдены (пустой контекст)
- `confidence = "low"`
- LLM не вернул валидный JSON после 3 попыток
- `sources` пуст → автоматически `needs_review=true`

---

## 🧠 Память/контекст

**Что хранится:**
- Документы разбиваются на чанки по 1000 символов с перекрытием 200 символов
- Для каждого чанка генерируется векторное представление (эмбеддинг) через модель `all-MiniLM-L6-v2`
- Эмбеддинги сохраняются в SQLite (BLOB) и индексируются через FAISS

**Как используется:**
1. При загрузке документа: текст → чанки → эмбеддинги → SQLite + FAISS
2. При вопросе: вопрос → эмбеддинг → FAISS поиск (топ-5) → контекст в LLM
3. LLM возвращает ответ с источниками → валидация → сохранение в qa_runs

**Преимущества векторного поиска:**
- Понимает смысл вопросов, а не просто совпадение слов
- Находит релевантные фрагменты даже при разных формулировках
- Работает локально, без облачных сервисов (FAISS)

---

## ✅ Контроль качества

### Валидация на входе
- `title` и `text` не пустые
- `title` ≤ 200 символов, `text` ≤ 500 000 символов
- `question` не пустой

### Валидация на выходе (LLM)
- ответ должен быть объектом (dict)
- обязательные поля: `answer`, `sources`, `confidence`, `needs_review`
- `answer` — строка
- `sources` — массив
- `confidence` — только `high`/`medium`/`low`
- `needs_review` — только boolean
- если `sources` пуст → автоматически `needs_review=true`

### Аудит
- каждое действие логируется в `audit_runs` с `duration_ms`
- при ошибке фиксируется текст ошибки
- статус: `ok` / `error` / `needs_review`

## ⚠️ Риски и меры снижения (минимум 5)

| № | Риск | Мера снижения |
|---|------|---------------|
| 1 | **Галлюцинации LLM** | Низкая температура (0.1), строгий JSON с валидацией, fallback на «данных недостаточно», ручная проверка |
| 2 | **Утечка данных / секретов** | API-ключ в `.env` (в `.gitignore`), SQLite локально, ключи не в репозитории |
| 3 | **Неточные ответы** | Обязательные источники (цитаты), метка `needs_review`, показ `document_id` |
| 4 | **Сбой LLM (недоступность, 401)** | 3 повторные попытки, безопасный fallback с `needs_review=true`, логирование ошибки |
| 5 | **Некорректный JSON от LLM** | regex-удаление markdown-обёртки, повторный парсинг, валидация структуры |
| 6 | **Перегрузка системы** | Ограничение `limit=5` для поиска, gunicorn с 2 воркерами |

## 🔧 Сложности и как решал

| Сложность | Решение |
|-----------|---------|
| **JSON-парсинг от LLM** | LLM иногда оборачивает ответ в markdown (```json ... ```), решено regex-удалением обёртки + 3 повторные попытки |
| **Поиск по ключевым словам** | Русский язык требует удаления стоп-слов, решено списком стоп-слов + сортировкой по количеству совпадений |
| **Векторный поиск** | Добавлен FAISS + sentence-transformers для семантического поиска вместо простого текстового |
| **Чанкирование больших документов** | Реализовано рекурсивное разделение по 1000 символов с перекрытием 200, лимит увеличен до 500К символов |
| **Импорт flask в VS Code** | Конфликт venv в родительской папке, решено выбором правильного интерпретатора и `.vscode/settings.json` |
| **Ленивая инициализация OpenAI** | Клиент создавался при импорте без ключа, решено через `get_client()` с отложенной инициализацией |
| **None от LLM** | `response.choices[0].message.content` может быть None, решено проверкой перед `.strip()` |
| **PyTorch Tensor vs NumPy** | `model.encode()` возвращает Tensor, решено преобразованием через `.numpy()` перед `.astype()` |
